二、XSS攻击

类似这样的代码<script>alert(123)</script>，在前端显示时，正常情况下若不修改Django设置，就不会被XSS攻击，也就是这串代码以字符串显示。

如果想让这串代码在网页出现alert一个123的效果，有两种方法：

  • （1）在前端自己加上 |safe
  • （2）在后端导入mark_safe模块之后mark_safe这串代码

但是，加safe时一定要小心！如果是可让用户填写的地方尽量不要加safe，要加就得在后端进行判断，把特殊字符过滤掉，只把安全的内容返回给前端。

三、跨站请求伪造

  • django为用户实现防止跨站请求伪造的功能，通过中间件django.middleware.csrf.CsrfViewMiddleware来完成。
  • CSRF是为了防止跨站请求伪造，通俗点说，就是用户在访问某网站时(GET请求)，该网站生成一个随机字符串，用户在提交数据时(POST请求)会带着这个随机字符串一并提交，如果和网站生成的随机字符串相同，才执行操作。
  • 要做csrf验证，就在form表单加上{% csrf_token %}。
  • jQuery.cookie 插件可以方便我们获取到cookie里的csrf_token，没有这个插件就需要用原生的方法获取到csrf_token的字符串，类似这样“csrf_token=jfieo53hwnjfklhnwoeutgh”，然后手动split拿到等号后面的值。

django中设置防跨站请求伪造功能有分为全局和局部：

  1. 全部禁用：注释掉中间件 ‘django.middleware.csrf.CsrfViewMiddleware’,
  2. 局部禁用：使用中间件，只在不想做验证的函数上面加@csrf_exempt，该函数则不需要csrf验证。需要导入from django.views.decorators.csrf import csrf_exempt
  3. 局部使用：注释掉中间件，只在想做验证的函数上面加@csrf_protect，该函数则需要csrf验证。需要导入from django.views.decorators.csrf import csrf_protect

六、Session

cookie是保存在客户端浏览器上的键值对；

七、中间件

  • django中的中间件（middleware），在django中，中间件其实就是一个类，在请求到来和结束后，django会根据自己的规则在合适的时机执行中间件中相应的方法。

  • 在django项目的settings模块中，有一个 MIDDLEWARE_CLASSES 变量，其中每一个元素就是一个中间件。

  • 中间件中可以定义5个方法，分别是：

    1. process_request(self,request)
    2. process_view(self, request, callback, callback_args, callback_kwargs)
    3. process_template_response(self,request,response)
    4. process_exception(self, request, exception)
    5. process_response(self, request, response)

    以上方法的返回值可以是None和HttpResonse对象，如果是None，则继续按照django定义的规则向下执行，如果是HttpResonse对象，则直接将该对象返回给用户。
